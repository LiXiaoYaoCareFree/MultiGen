MultiGen 系统通过 **A2A（Agent-to-Agent）协议**和 **MCP（Model Context Protocol）协议**构建了双轨并行的工具集成架构，前者实现了 Agent 之间的协作通信，后者提供了标准化的外部工具接入能力。这两种协议共同构成了系统扩展性的核心支柱，使得单个 Agent 能够突破自身能力边界，通过工具调用和网络协作完成复杂任务。

## A2A 协议：Agent 间协作的通信标准

A2A 协议定义了一套标准化的 Agent 间通信机制，允许一个 Agent 将子任务委托给远程 Agent 执行。系统中的 `A2AClientManager` 充当协议实现的核心组件，负责管理多个远程 Agent 的连接生命周期和元数据缓存。该管理器在初始化阶段通过 HTTP 请求获取所有已配置 Agent 的卡片信息，这些卡片遵循标准的 JSON 结构，包含 Agent 标识、技能描述、调用端点等元信息。缓存机制避免了每次工具调用时重复获取卡片数据的网络开销，这是高并发场景下性能优化的关键设计。

协议的调用流程遵循 RPC 风格的消息传递模式。当本地 Agent 需要调用远程 Agent 时，管理器会构造一个包含标准 JSON-RPC 字段的请求体，使用 `message/send` 方法将任务查询发送到目标 Agent 的端点。请求结构包含自动生成的 UUID 作为消息标识符、固定的 JSON-RPC 版本声明以及符合协议规范的消息体结构，其中消息体封装了用户查询的文本内容。这种标准化格式确保了不同实现之间的互操作性，使得 MultiGen 能够与任何兼容 A2A 协议的外部系统集成。Sources: [a2a.py](api/app/domain/services/tools/a2a.py#L13-L60)

从架构设计的角度观察，A2A 工具的实现体现了资源管理的精细考量。系统使用 `AsyncExitStack` 管理异步上下文以避免深层嵌套的 `with` 语句，这个设计模式在管理多个异步资源时显著提升了代码的可读性和可维护性。管理器同时维护了 HTTP 客户端实例和 Agent 卡片字典，在清理阶段通过 `aclose()` 方法确保所有网络连接和相关资源被正确释放，防止长时间运行的服务进程出现资源泄漏。`A2ATool` 类进一步封装了管理器，通过装饰器模式声明工具的参数模式，使得 Agent 可以像调用本地工具一样调用远程 Agent。Sources: [a2a.py](api/app/domain/services/tools/a2a.py#L65-L206)

## MCP 协议：标准化的工具接入框架

MCP 协议作为 Anthropic 推动的开放标准，为 LLM 应用提供了统一的工具集成接口。MultiGen 中的 `MCPClientManager` 实现了完整的多传输协议支持，能够同时管理基于标准输入输出（stdio）、服务器发送事件（SSE）以及流式 HTTP 的 MCP 服务器连接。这种多协议支持使得系统能够灵活集成不同类型的工具，从本地命令行工具到远程 Web 服务，无需修改核心代码。配置文件 `mcptools.json` 定义了服务器列表和启动参数，系统在运行时动态解析这些配置并与 MCP 服务器建立连接。

协议实现的技术细节揭示了资源优化策略。管理器在初始化每个 MCP 服务器连接后，会立即调用 `list_tools()` 方法获取服务器提供的所有工具及其参数模式，并将这些信息缓存到字典中。这种设计避免了每次工具调用时重复查询工具列表的网络往返，对于提供大量工具的 MCP 服务器尤为重要。工具名称经过清理函数处理以确保符合 OpenAI 的命名规范，该函数移除非法字符、处理数字开头、截断过长名称，保证工具声明能够被 LLM 正确解析和调用。Sources: [mcp.py](api/app/domain/services/tools/mcp.py#L1-L200)

连接不同传输协议需要差异化的技术实现。对于 stdio 协议，系统创建子进程运行指定的命令，通过标准输入输出流与子进程通信；对于 SSE 协议，建立长连接 HTTP 连接接收服务器推送的事件流；对于 streamable_http 协议，使用支持流式响应的 HTTP 客户端。所有这些连接都通过 `AsyncExitStack` 进行统一的生命周期管理，确保在服务关闭时所有连接和会话被正确清理。每个 MCP 服务器连接对应一个 `ClientSession` 实例，该实例提供了工具调用、资源读取等标准 MCP 操作的抽象接口。Sources: [mcp.py](api/app/domain/services/tools/mcp.py#L201-L426)

## 双协议架构的设计平衡

A2A 和 MCP 两种协议的并存并非功能重复，而是解决了不同层面的扩展需求。MCP 专注于单一工具的功能调用，适合集成搜索引擎、数据库访问、文件操作等原子能力；A2A 则实现了 Agent 层面的协作，允许将复杂任务委托给具备专业能力的远程 Agent。这种区分在 `agent_service.py` 的依赖注入中清晰体现：两个协议的配置对象被独立传递给任务执行器，执行器根据任务需求动态选择合适的工具或协作对象。系统的配置文件分离也反映了这种关注点分离的设计哲学。Sources: [agent_service.py](api/app/application/services/agent_service.py#L1-L100)

从错误处理和容错性的角度比较，两种协议展现出不同的策略。A2A 管理器在初始化时即使部分 Agent 服务器不可用也会继续加载其他可用服务，通过日志记录失败项而非中断整个初始化流程。MCP 管理器采用类似策略，单个 MCP 服务器连接失败不会阻止其他服务器的连接尝试。这种健壮性设计确保了系统的可用性不会因单个外部服务的故障而受影响。工具调用的返回值统一封装为 `ToolResult` 结构，包含成功标志、消息和可选的数据字段，使得上层业务逻辑能够以一致的方式处理两种协议的调用结果。Sources: [a2a.py](api/app/domain/services/tools/a2a.py#L62-L90) [mcp.py](api/app/domain/services/tools/mcp.py#L85-L120)

协议配置的动态性支持多租户场景。A2A 和 MCP 的配置并非硬编码，而是在 Agent 服务初始化时通过参数注入。这意味着不同的用户会话或任务可以使用不同的工具集和协作 Agent 集合，为未来的多租户隔离和权限控制提供了架构基础。配置验证逻辑确保只有启用状态的服务才会被加载到运行时环境，前端界面可以展示所有已配置的服务状态，而执行中的 Agent 仅能访问已启用的工具和远程 Agent，实现了配置可见性与执行可用性的分离。Sources: [agent_service.py](api/app/application/services/agent_service.py#L25-L45)

## 协议对比与适用场景

下表概括了两种协议的核心差异，帮助开发者根据具体需求选择合适的扩展方式：

| 对比维度 | A2A 协议 | MCP 协议 |
|---------|---------|---------|
| **核心定位** | Agent 间协作与任务委托 | 标准化工具函数集成 |
| **通信协议** | HTTP/JSON-RPC | stdio/SSE/streamable_http |
| **调用目标** | 具备完整能力的远程 Agent | 单一功能的工具函数 |
| **资源粒度** | Agent 级别（包含状态、记忆） | 工具级别（无状态执行） |
| **配置位置** | config.yaml 中的 a2a_servers | mcptools.json 中的 mcpServers |
| **元数据缓存** | Agent Card（技能、端点） | Tool Schema（参数模式） |
| **适用场景** | 复杂任务分解、专业能力互补 | 原子操作执行、外部服务调用 |
| **初始化开销** | 网络请求获取 Agent Card | 工具列表查询、会话建立 |

两种协议在实际应用中常常协同工作。典型的任务流程可能涉及：Agent 首先通过 MCP 工具搜索相关信息，然后调用 A2A 协议委托专业 Agent 进行深度分析，最后再使用 MCP 工具存储结果或发送通知。这种工具与协作的组合使得 MultiGen 能够处理从简单查询到复杂决策支持的广泛场景，而无需在核心 Agent 逻辑中硬编码具体的能力实现。

## 技术实现的关键细节

资源管理的正确性是协议实现的底层保障。两个 ClientManager 都实现了 `cleanup()` 方法，在方法中调用 `AsyncExitStack.aclose()` 关闭所有通过该栈管理的异步上下文，这包括 HTTP 客户端连接、子进程句柄、SSE 连接等。清理顺序遵循后进先出的原则，确保依赖关系正确的资源释放。缓存的工具元数据和 Agent 卡片在清理时被显式清空，避免内存泄漏。`initialized` 标志位防止重复初始化，这是高并发环境下保护资源的关键机制。Sources: [a2a.py](api/app/domain/services/tools/a2a.py#L143-L166) [mcp.py](api/app/domain/services/tools/mcp.py#L315-L340)

工具注册机制采用装饰器模式简化声明。`@tool` 装饰器允许开发者以函数注解的方式定义工具名称、描述和参数模式，装饰器内部自动提取这些元数据并注册到工具管理器。这种设计使得工具定义保持简洁和可读，同时确保生成的工具声明符合 LLM 的函数调用规范。A2A 工具提供了 `get_remote_agent_cards` 和 `call_remote_agent` 两个主要接口，前者用于发现可用 Agent，后者用于执行远程调用。MCP 工具的注册更为动态，所有从 MCP 服务器获取的工具自动转换为可调用对象，无需手动编写包装代码。Sources: [a2a.py](api/app/domain/services/tools/a2a.py#L168-L206)

环境变量的合并策略体现了系统集成的灵活性。对于 stdio 类型的 MCP 服务器，系统将传递的环境变量与 `os.environ` 合并后注入到子进程环境，这使得 MCP 服务器能够访问系统级的环境配置，同时允许针对特定服务器配置独立的环境变量。这种设计在访问密钥管理、调试标志控制等场景中特别有用，开发者可以为不同的 MCP 服务器配置独立的认证凭证，而无需修改全局环境。Sources: [mcp.py](api/app/domain/services/tools/mcp.py#L140-L180)

理解这两种协议的工作机制是掌握 MultiGen 扩展性的关键，建议接着阅读 [Agent 服务实现](13-agent-fu-wu-shi-xian) 以了解这些协议如何在完整的任务执行流程中被调度使用。
